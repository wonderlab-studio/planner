## Архитектура (мульти-пользовательская, v2)

### Поток данных

```
users.json / env → load_users() → list[UserConfig]
  ↓ bot.py main()
для каждого user:
  KaitenClient(board_id, lane_id, token, base_url)  # per-user
  setup_board(client, user) → column_ids    # пропуск если user.column_ids задан
  BoardLogic(client, column_ids)            # per-user
  MorningLogic(client, logic)              # per-user
  Notifier(chat_id)                         # per-user
  → UserSchedulerCtx, UserHandlerCtx

Scheduler(users=[...], claude)             # один на всех, итерирует
HandlersConfig(users={chat_id: ctx}, claude)  # роутинг по chat_id
```

### Ключевые файлы

| Файл | Ответственность |
|---|---|
| user_config.py | UserConfig dataclass, load_users(), REQUIRED_COLUMN_NAMES |
| board_setup.py | Идемпотентная настройка доски, alias-маппинг имён |
| kaiten_client.py | HTTP-клиент, KaitenClient(board_id, lane_id, token, base_url) |
| board_logic.py | BoardLogic(client, column_ids), WEEKDAY_COLUMNS |
| morning_logic.py | Алгоритм утра v4, MorningLogic(client, logic) |
| scheduler.py | APScheduler джобы, UserSchedulerCtx, мульти-user цикл |
| handlers.py | Telegram handlers, HandlersConfig, UserHandlerCtx |
| notifier.py | Notifier(chat_id) — отправка без апдейта |
| db.py | SQLite флаги, daily_flags(date, user_id) |

## Формат разделителей (КРИТИЧНО)

Разделители — карточки с `blocked=True`. Формат:
```python
title = "-"                              # НЕ название секции!
block_reason = "Утро" | "День" | "Вечер" | "На контроле"
blocked = True
```

Разделители существуют ТОЛЬКО в колонках дней (Пн–Вс). Не в "Следующая неделя".

## Безопасность: секреты всегда в env

**КРИТИЧНО: никогда не хранить токены/пароли напрямую в конфиг-файлах (users.json и т.д.).**

Паттерн: в конфиге хранится только **имя** env-переменной, значение читается при старте.

```json
// users.json — ПРАВИЛЬНО
{
  "kaiten_token_env": "KAITEN_TOKEN_ALICE",
  "kaiten_base_url_env": "KAITEN_BASE_URL_ALICE"
}
```

```python
# user_config.py — разрешение ссылки при загрузке
token_env = item.get("kaiten_token_env")        # "KAITEN_TOKEN_ALICE"
kaiten_token = os.getenv(token_env) if token_env else None  # реальный токен из env
```

Правило: **значение секрета** живёт только в env (Railway Variables / .env). В любом файле, который попадает в git или передаётся между агентами, хранится лишь имя переменной.

## board_setup.py — логика настройки доски

При старте: если `user.column_ids` не пустой → board_setup пропускается полностью.

**Alias-маппинг (`_NAME_ALIASES`):** нестандартные имена → стандартные.
Например, "Пн" → "Понедельник". Alias всегда перезаписывает точное совпадение.

**Идемпотентность:**
- Если колонка/разделитель уже есть → не создаёт дубль
- Удаляет лишние колонки ТОЛЬКО если они ПУСТЫЕ
- Дублирующие стандартные колонки (после alias-маппинга) — удаляются если пустые

**Пропуск setup:** если `user.column_ids` задан в users.json → setup не вызывается.
Это предотвращает случайное создание дублей на уже настроенных досках.

## Особенности Kaiten API (поведение из продакшена)

- КРИТИЧНО: `blocked=True` в теле `POST /cards` **игнорируется Kaiten API** — карточка создаётся разблокированной. Для создания разделителя (blocked card) нужен отдельный `PATCH /cards/{id}` с `{"blocked": True, "block_reason": "..."}` после POST.
- КРИТИЧНО: теги добавляются ТОЛЬКО через `POST /cards/{id}/tags` с телом `{"name": "tagname"}`. Поле `tag_ids` в PATCH молча игнорируется — теги не появятся.
- КРИТИЧНО: `archive_card` вызывать только через `BoardLogic.archive_card(card_id)`. `KaitenClient.archive_card` требует явного `archive_column_id` — только `BoardLogic` его знает. Прямой вызов через `user_ctx.kaiten.archive_card(...)` из handlers/scheduler — ошибка.
- КРИТИЧНО: при добавлении нового метода в `KaitenClient` (документируешь в docs/interfaces.md) — реализовать метод в `kaiten_client.py` в том же PR. Ссылка на несуществующий метод из handlers/scheduler вызовет AttributeError в рантайме.

## Алгоритмы размещения по времени: occupied-список vs курсор

В `morning_logic.py` (`_BlockScheduler`) размещение задач по временным блокам держит ДВА разных
механизма: список занятых интервалов (`_occupied`, источник истины — что реально нельзя занимать
повторно) и курсор (`_cursor`, эвристика — с какой точки блока начинать ПОИСК свободных
интервалов для очередной задачи/группы). Это разделение легко упустить: кажется, что курсор тоже
защищает от наложений, но на самом деле наложения защищает только `_occupied`.

Проблема, к которой это приводит: если задача атомарная («не дробить») или сегментированная
пропускает слишком маленький свободный промежуток (не влезает целиком / короче минимального
сегмента) и размещается ПОЗЖЕ пропущенного промежутка, курсор всё равно продвигается вперёд —
и пропущенный промежуток становится недоступен для всех последующих групп приоритета, хотя они
вполне могли бы его заполнить (`_occupied` не помешал бы — он про этот промежуток ничего не знает).

Решение без усложнения структуры данных: сбрасывать курсор к началу блока между группами
приоритета (`reset_cursor_to_start()`), НЕ трогая `_occupied`. Это безопасно всегда — в обычном
случае (группа ничего не пропустила) сброс даёт идентичный результат, потому что `_occupied` и
так вычтет уже занятые интервалы; выигрыш проявляется только в случае пропуска, когда более
поздняя (менее приоритетная) группа получает шанс заполнить пропущенное пространство.

**Общий принцип для будущих правок в подобных алгоритмах:** если структура данных совмещает
«источник истины о занятости» (список/множество интервалов) и «эвристику порядка обработки»
(курсор/указатель) — при появлении многоэтапной/многоприоритетной обработки первым делом проверяй,
не блокирует ли эвристика доступ к пространству, которое источник истины по-прежнему считает
свободным. Если блокирует — эвристику можно сбрасывать между этапами без риска, потому что
источник истины не даст создать реальное наложение.


тогда board_setup будет пропущен.

## Протокол деплоя

1. Оркестратор обновляет `docs/requirements.md` (новые фичи/изменения)
2. Параллельно вызывает `deploy` и `retro` агентов:
   - `deploy` — проверяет импорты → коммитит → пушит (или возвращает BLOCKED)
   - `retro` — получает список ошибок сессии → обновляет документацию
3. Если deploy вернул BLOCKED → оркестратор делегирует правку нужному субагенту → повторяет деплой

## Шаблон промпта для субагента

Обязательные разделы при постановке задачи реализующему агенту:

```
**Задача:** [что именно изменить — конкретно, не "улучши код"]

**Файлы для изменения:** [перечислить явно]
**Файлы НЕ трогать:** [перечислить явно]

**Контекст:** [ключевые выдержки из requirements/interfaces ИЛИ findings от Explore агента]

**Требования к реализации:** [паттерны, ограничения, на что обратить внимание]

```

### Масштабные задачи (10+ пунктов): Explore-граф перед реализацией

Когда пользователь даёт большой список (10+ пунктов) с багами, фичами и архитектурными гипотезами — обязательная последовательность **до запуска первого реализующего агента**:

1. Сгруппировать по файлам: какие пункты затрагивают один и тот же файл?
2. Запустить **параллельные Explore-агенты** — по одному на смысловую группу (Telegram, Kaiten/scheduler, и т.д.). Каждый возвращает точные цитаты кода с номерами строк.
3. Построить **полный граф задач** (TaskCreate + `blockedBy`) для всех пунктов, затрагивающих один файл — весь граф сразу, до первой делегации.
4. Если Explore выявил расхождение между тем, как пользователь описывает баг, и тем, что показывает код — объяснить расхождение явно в промпте реализующего агента, не молча «исправлять что кажется правильным».

### Точность промптов: якоря для агентов без поиска

Субагенты этого проекта имеют Read, Write и Edit (без Bash/Grep). Они не могут самостоятельно найти нужную строку без полного чтения файла. Если оркестратор читал нужные файлы перед постановкой задачи — включи в промпт точные номера строк и текущие фрагменты кода «as is» для каждого места изменения. Без явных якорей агент вынужден читать весь файл целиком и восстанавливать контекст самостоятельно — это медленнее и увеличивает риск ошибочной правки соседних строк.

### Парные точки входа: требуй зеркальную логику явно

Когда одно и то же действие (перенос, подтверждение риска, обновление поля) реализуется через ДВЕ точки входа — кнопочный диалог (`ConversationHandler`) и текстовую команду (`_handle_*`) — явно называй обе точки входа в задании субагенту и требуй одинакового поведения в каждой. Не описывай одну как «образец»: агент не выведет вторую самостоятельно. Типичные парные пары в этом проекте: `received_move_target_cb` / `_handle_move`, диалог создания / команда `создать`.

Правило шире пары «две функции»: оно применяется и к двум условным веткам ВНУТРИ одной функции, если обе ведут к одному побочному эффекту (например, перенос карточки), но реализованы независимо (fast-path для коротких задач и обычный путь для длинных). При постановке задачи субагенту — явно называй ОБЕ ветки, не рассчитывай, что агент вспомнит про вторую после исправления первой.

### Эстафета точных имён: последовательные волны по одному файлу

Когда Tier 3 задача бьётся на N последовательных волн по одному файлу (T1→T2→…→TN) — каждый промпт следующей волны включает явный блок «Контекст из предыдущей волны» с именами функций, ключей `user_data`, строк `callback_data` и состояний `ConversationHandler`, скопированными **дословно** из раздела «Интерфейс» отчёта предыдущего агента — не по памяти оркестратора.

Агент не имеет доступа к тому, что добавила предыдущая волна; без явной передачи он читает весь файл заново и восстанавливает контекст — медленнее и с риском потери деталей.

Следствие для шаблона ответа: раздел «Интерфейс» обязателен всегда — не только при изменении публичных методов. Новые ключи `user_data`, паттерны `callback_data`, имена состояний `ConversationHandler` — тоже интерфейс для следующей волны.

### Когда статический анализ бага не даёт результата: эскалация к логированию, а не угадывание

Если оркестратор (сам и/или через одного реализующего субагента) дважды перечитал код и не нашёл
причину подтверждённого пользователем бага — не стоит делать третий заход «на глаз» или
спекулятивно менять логику в надежде «а вдруг это оно». На этом этапе более высокая отдача —
добавить точечное диагностическое логирование прямо в месте принятия решения (все переменные,
влияющие на исход: сырые значения полей ДО парсинга и результат ПОСЛЕ), закоммитить это отдельно
от остальных фиксов и явно сообщить пользователю, что причина не найдена, а логирование добавлено
для следующего воспроизведения. Ложная уверенность («кажется, я нашёл») хуже честного «не нашёл,
жду данные из прода» — особенно когда живого доступа к продакшен-окружению (Kaiten API ответы,
Railway-логи) у агента нет и первопричину нельзя подтвердить иначе как по факту повторения.

### Лимит вывода модели: диагностика и предотвращение

Ошибка `output token maximum` у субагента — отдельный паттерн отказа, не «агент завис» и не
«логическая ошибка». Диагностика в порядке приоритета:

1. **Есть ли у агента инструмент `Edit`?** Агент с только Read+Write вынужден переписывать файл
   целиком через Write. На файлах ~500–800+ строк (handlers.py в этом проекте достигает ~2900
   строк) любая задача с добавлением ~150+ строк кода при наличии рассуждений модели превышает
   лимит вывода. Добавить Edit в конфигурацию агента — это снимает проблему системно: агент
   пишет только дифф, не весь файл.

2. **Если Edit уже есть — задача слишком большая.** При первом столкновении с лимитом дробить
   минимум в 4–5 раз мельче, не в 2 раза. Деление «в 2 раза» часто снова упирается в лимит:
   рассуждения модели и контекст задачи занимают значительную долю вывода независимо от размера
   кода. Каждый лишний цикл «деплой → падение → пересборка» стоит раунда ожидания.

**Правило по умолчанию:** реализующие агенты (kaiten-agent, scheduler-agent, telegram-agent,
claude-api-agent) должны иметь `Edit` в списке tools наряду с Read и Write.

## Шаблон ответа от субагента

Субагент завершает работу и возвращает структурированный отчёт:

```
**Сделано:** [список изменений с именами файлов и сутью правок]
**Пропущено:** [что не менялось и почему — если были отклонения от задачи]
**Интерфейс:** [изменились ли публичные методы/сигнатуры — важно для оркестратора]
**Проверка:** [результат синтаксис-чека или "не запускал — нет доступа к Bash"]
**Блокеры:** [проблемы требующие решения оркестратором, или "нет"]
```

**Верификация владения перед ограничениями в промпте:** перед тем как запрещать или разрешать агенту трогать конкретный файл — сверься с таблицей выше. Типичная ошибка: scheduler-agent вызывает метод из notifier.py, поэтому кажется что «он работает с этим файлом» — но реализовывать notifier.py должен telegram-agent. Запрет правильному владельцу и разрешение без реализации — источник ошибок на стыке волн.
