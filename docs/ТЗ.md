## P0 — блокирует лояльность (боль названа несколькими персонами)

### 1. Пересборка плана в течение дня («пересобрать»)

**Проблема.** Утренний алгоритм (`morning_logic.py`) запускается один раз и расставляет
все задачи по слотам 09:00–19:00 / 19:00–22:00. Если день пошёл не по плану (внеплановая
встреча, проспанное утро) — расписание не пересчитывается до следующего утра.

**Ключевая сложность.** После утреннего прогона у ВСЕХ карточек колонки уже выставлен
`event_time` — и у «жёстких» событий (встречи, регулярные задачи с явным временем —
Фаза 1), и у «мягких» (авторазмещённых алгоритмом — Фаза 3). После прогона они
неразличимы по данным карточки: оба типа имеют `event_time` на сегодня. А пересборка
обязана трогать только вторые — иначе бот будет переставлять реальные встречи.

**Решение — маркер «жесткое событие».**
Заводим тег `жесткое событие` (создаётся автоматически при первом использовании через
`add_tag_by_name`, как и остальные теги — ручной настройки в Kaiten не требует).

- Фаза 0 - поправка - добавляем в пул также карточки из сегодняшней колонки с Событием на сегодня. Удаляем из всех карточек пула тэг "жесткое событие" везде, где он есть.
- Фаза 1 (жёсткие события) ставим на такие события тэг "жесткое событие" при резервировании временного слота. 
- При добавление карточки через команду чатбота на сегодня на определенное время - этой карточке присваивается тэг "жесткое событие". При этом чат предлагает пользователю пересобрать расписание. Если пользователь соглашается - запускается пересборка расписания (replan).

**Алгоритм `MorningLogic.replan(now: datetime) -> list[Card]`** (новый публичный метод):

1. Загрузить карточки сегодняшней колонки (`self._logic.get_today_column_id()`), определить секцию каждой через разделители (переиспользовать `_get_card_section`).
2. Разбить не-разделители, не-архивные карточки на 4 группы:
- **«На контроле»** — не трогаем вообще.
- **прошедшие/идущие** — `card.event_time <= now` — не трогаем (уже случились или начались; трогать «на живую» опасно).
- **жёсткие будущие** — `event_time > now` и есть тэг `жесткое событие` — это встречи; не двигаем, но **резервируем** их слот в планировщике (аналогично Фазе 0).
- **кандидаты на пересборку** — `event_time > now` и тега `жесткое событие` НЕТ, ИЛИ `event_time is None` (например, карточка создана командой «создать» посреди дня без явного времени).
3. Собрать `work_sched` / `evening_sched` (`_BlockScheduler`) с границами блоков как
   обычно (09:00–19:00 / 19:00–22:00), но курсор `_cursor` выставить в
   `max(now_rounded_to_5min, _WORK_START)` вместо 09:00. Если `now` уже после 19:00 —
   работаем только с `evening_sched`, а если после 22:00 — сразу все кандидаты в overflow.
4. Зарезервировать в схедулерах слоты «жёстких будущих» карточек (как Фаза 0).
5. Прогнать кандидатов через **ту же классификацию по 9 группам**, что и в
   `_schedule_today` (эту часть стоит вынести в приватный метод
   `_classify_groups(cards, today) -> list[list[Card]]`, чтобы не дублировать код
   между `_schedule_today` и `replan`).
6. Прогнать через существующую Фазу 3 (сегментированное размещение,
   `try_place_segmented`, `_set_event_time`, `_move`) — тоже стоит вынести в
   переиспользуемый метод `_place_groups(processing_order, today, today_col_id, preloaded)`.
7. Не влезшее — через существующий `_handle_overflow` (без изменений, п. 3 Прозрачность overflow ниже
   добавит туда отчёт и логику пропуска выходных).
8. Вернуть свежие карточки колонки, как и `_schedule_today`.

**Рефакторинг `_schedule_today`:** текущее тело Фазы 2+3 (строки ~412–579) выносится в
`_classify_groups` + `_place_groups`, чтобы `_schedule_today` и `replan` использовали
общий код с разными входными кандидатами и разным стартовым курсором.

**Хендлер (`handlers.py`):**
- Новый текстовый паттерн: добавить `|пересобрать|перепланируй` в `_MAIN_COMMANDS_FILTER`
  (строка 50–53), в ветки `text_handler` и `conv_fallback_text_cb`.
- Новая функция `_handle_replan(update, user_ctx)`:
  ```
  await update.message.reply_text("🔄 Пересобираю план с текущего момента…")
  cards = await user_ctx.replan_routine()
  # сгенерировать план тем же способом, что и handle_morning (Claude.generate_morning_plan),
  # приложить блок overflow из last_overflow (см. п.3), отправить + send_card_buttons
  ```
- `UserHandlerCtx` (dataclass) — добавить поле `replan_routine: RoutineCallable`.
- `bot.py` — в блоке создания `users_handler` добавить:
  ```python
  async def replan_routine(_sc=sched_ctx) -> None:
      await scheduler.run_replan_for_user(_sc)
  ```
- `scheduler.py` — новый метод `Scheduler.run_replan_for_user(user_ctx)`: вызывает
  `user_ctx.morning.replan(_now_msk())`, генерирует план через `self._claude.generate_morning_plan`
  (переиспользуя `_card_to_dict`/`_extract_section_cards`), шлёт через
  `user_ctx.notifier.send_morning_plan`. **Не трогает** флаг `morning_done` — пересборка
  разрешена сколько угодно раз за день.

**Файлы:** `morning_logic.py` (новый `replan`, рефакторинг Фазы 2/3 в переиспользуемые

## P1 — конкретный инцидент, весомая боль

### 4. Защита регулярных карточек от архивации через «Готово»

**Проблема.** «Готово» вызывает `BoardLogic.archive_card` → перемещение в колонку
Архив. Для карточки с тегом `ежедневно`/`по будням`/`по выходным`/`еженедельно` это
означает, что она пропадает из ротации: следующее утро её не увидит ни во «вчерашней»,
ни в «сегодня-stale» выборке (обе смотрят только на колонки дней недели, не на Архив).
Через 30 дней `run_archive_cleanup_job` (`scheduler.py:398`) её физически удалит.
Возвращаемое поведение регулярных задач основано на **той же самой карточке**,
переносимой день в день — архивация её убивает навсегда.

**Решение — «мягкое» завершение для регулярных карточек.**
карточка при завершении должна идти не в архив, а в следующий подходящий день

### 6. Пагинация кнопок карточек (лимит 20)

**Проблема.** `send_card_buttons` (`handlers.py:203–241`) жёстко берёт
`task_cards[:MAX_CARD_BUTTONS]` (20), остальное не показывает вообще.

**Решение — постраничный вывод:**
```python
async def send_card_buttons(cards, bot, chat_id, page: int = 0) -> None:
    task_cards = [c for c in cards if not c.blocked and not c.archived]
    if not task_cards:
        return
    start = page * MAX_CARD_BUTTONS
    shown = task_cards[start : start + MAX_CARD_BUTTONS]
    keyboard = [[InlineKeyboardButton(..., callback_data=f"card:{c.id}")] for c in shown]

    nav_row = []
    if page > 0:
        nav_row.append(InlineKeyboardButton("← Назад", callback_data=f"page:{page-1}"))
    if start + MAX_CARD_BUTTONS < len(task_cards):
        nav_row.append(InlineKeyboardButton("Ещё →", callback_data=f"page:{page+1}"))
    if nav_row:
        keyboard.append(nav_row)

    markup = InlineKeyboardMarkup(keyboard)
    total_pages = (len(task_cards) - 1) // MAX_CARD_BUTTONS + 1
    text = f"📋 *Карточки на сегодня* (стр. {page+1}/{total_pages}):"
    await bot.send_message(chat_id=chat_id, text=text, parse_mode=ParseMode.MARKDOWN, reply_markup=markup)
```
- Новый top-level `CallbackQueryHandler` (регистрируется в `build_handlers`, **вне**
  `conv_handler` — это не часть диалога выбора карточки, доступен всегда):
  ```python
  async def page_nav_cb(update, context) -> None:
      query = update.callback_query
      await query.answer()
      chat_id = update.effective_chat.id
      user_ctx = cfg.users.get(chat_id)
      if user_ctx is None:
          return
      page = int(query.data.split(":")[1])
      today_cards = await user_ctx.kaiten.get_cards(user_ctx.logic.get_today_column_id())
      try:
          await query.delete_message()
      except Exception:
          pass
      await send_card_buttons(today_cards, context.bot, chat_id, page=page)

  app.add_handler(CallbackQueryHandler(page_nav_cb, pattern=r"^page:\d+$"))
  ```
  Регистрировать **после** `conv_handler`, но `pattern` не пересекается с `card:\d+`/
  `action:\w+`, так что порядок не критичен.
- `_resend_card_buttons` (строка 587) — вызывает `send_card_buttons(..., page=0)` явно
  (после любого действия над карточкой всегда возвращаемся на первую страницу).

**Файлы:** `handlers.py` (`send_card_buttons` — параметр `page`, новый `page_nav_cb`,
регистрация хендлера, `_resend_card_buttons`).

### 8. Мягкое сопротивление при переносе критики с близким дедлайном

- В `received_hours_cb` (отложить, п.5) и `received_move_target_cb` (перенос) — перед

  ```python
  if card.importance == "критическое" and due_soon:
      context.user_data["pending_move"] = {"target_col_id": target_col_id, "section": section}
      await update.message.reply_text(
          f"⚠️ «{title}» — критическая задача с дедлайном {due_date}.\n"
  ```

### 9. Флаг «не дробить» + минимальный размер сегмента

Две отдельные, но связанные правки.

**9a. Пропуск вопроса «Сколько часов?» для коротких задач.**
`action_today_cb` (`handlers.py:720`) сейчас безусловно спрашивает часы. Меняем:
```python
async def action_today_cb(update, context) -> int:
    query = update.callback_query
    await query.answer()
    chat_id = update.effective_chat.id
    user_ctx = cfg.users.get(chat_id)
    card_id = context.user_data.get("selected_card_id")
    card = await user_ctx.kaiten.get_card(card_id)
    if card and card.size is not None and card.size <= 0.25 and card.size != 999:
        # короткая привычка — не спрашиваем часы, сразу переносим с тем же размером
        await _postpone_card(user_ctx, card, hours=None)
        ...
        return ConversationHandler.END
    await query.edit_message_text("⏱ Сколько часов займёт задача?", ...)
    return AWAITING_HOURS
```
Логика переноса (сейчас — второй половина `received_hours_cb`, строки ~886–920) должна
быть вынесена в переиспользуемую функцию `_postpone_card(user_ctx, card, hours)`, чтобы
не дублировать между `action_today_cb` (короткий путь) и `received_hours_cb` (обычный путь).

**9b. Минимальный размер сегмента + тег «не дробить».**
- Новый тег `не дробить` (создаётся автоматически через `add_tag_by_name`, как и
  `авторазмещено`).
- `_BlockScheduler` (`morning_logic.py:75–164`) — новый метод:
  ```python
  def try_place_atomic(self, duration_min: int) -> list[tuple[int, int]] | None:
      """Как try_place_segmented, но НЕ дробит — ищет один непрерывный интервал."""
      for seg_s, seg_e in self.free_intervals_from_cursor():
          if seg_e - seg_s >= duration_min:
              end = seg_s + duration_min
              self._occupied.append((seg_s, end))
              self._occupied.sort()
              self._cursor = end
              return [(seg_s, end)]
      return None  # не влезает целиком нигде — overflow
  ```
- В Фазе 3 (`_place_groups`, после рефакторинга из п.1) — выбор метода размещения:
  ```python
  place_fn = sched.try_place_atomic if _TAG_NO_SPLIT in card.tag_ids else sched.try_place_segmented
  segments = place_fn(dur_min)
  ```
- Минимальный размер сегмента (чтобы не создавать «часть 2/2 — 15 минут» даже без тега):
  в `try_place_segmented`, при нарезке — если остаток свободного интервала после чанка
  меньше `_MIN_SEGMENT_MIN = 15`, либо забирать его целиком в этот чанк (слегка превышая
  нужную длительность), либо пропускать интервал целиком, если он меньше минимума.

**Файлы:** `handlers.py` (`action_today_cb`, `_postpone_card` — общий хелпер),
`morning_logic.py` (`_BlockScheduler.try_place_atomic`, минимальный сегмент в
`try_place_segmented`, выбор метода в Фазе 3), `kaiten_client.py` (тег через
`add_tag_by_name`, ничего менять не нужно — тег создаётся по имени на лету).

Чат запрашивает комментарий при закрытии задач. "-" воспринимается как "не надо добавлять комментарий". Поменяй на ".", т.к. точку быстрее ввести с телефона.

### 14. Всегда доступные кнопки в нужном порядке

Сейчас после утреннего расписания в чате не появляются кнопки. чтобы они появились, надо написать Утро - это костыльный метод. Надо чтобы кнопки появлялись после утренней логики и после пересборки. Также надо чтобы порядок кнопок совпадал с порядком задач в расписании.

### 15. Создание задач с размером

Надо чтобы была возможность при создании задачи командой указать размер, добавив фразу "размер - 2" или " размер 0,5" или "размер - 0.5"

### 16. Странный раздел в описании расписания утром

В сообщении с расписанием после утренней логики много дней внизу были фразы типа "задач без времени - нет", не очень понятно. Похоже что у claude какая-то лишняя инструкция и он ищет какие-то задачи без назначенного времени или вроде того

### 17. Не работает чистка архива по понедельникам

Нужно определять карточки для чистки так: они находятся в колонке "Архив" и у них "last_moved_at": более месяца назад. Такие карточки надо удалять насовсем методом delete

### 18. Новый пользователь с другого аккаунта

Планирую добавить пользователя с другого аккаунта Kaiten и у меня есть подозрения, что это может не сработать. Причина: кастомные поля карточки в kaiten и метки. Кастомные поля вроде "День недели" "Событие" "Важность" - будут отсутствовать у пользователя, их надо создать. Сможет ли наш сервис создать их автоматически?
Метки же будут с другими id, нежели у меня.
Нужно обдумать это, проверить логику, и если есть проблемы - решить их.
