# Требования системы личного планирования

## 1. Структура канбан-доски (Kaiten)

### Колонки

| Название | Назначение |
|---|---|
| Понедельник – Воскресенье | Задачи текущей недели |
| Следующая неделя | Задачи на следующую неделю |
| Далекие времена | Задачи без конкретного срока |
| Долгий ящик | Отложенные на неопределённо долго |
| Архив | Выполненные задачи |

### Разделители внутри колонки дня

Каждая колонка дня разделена на секции карточками-заголовками (`blocked=True`, секция из `block_reason`):

| Секция | Смысл |
|---|---|
| **Утро** | Срочные и приоритетные задачи — выполняются первыми, вне зависимости от времени суток |
| **День** | Основные задачи дня |
| **Вечер** | Задачи с тегом «вечерняя» |
| **На контроле** | Задачи, которые пользователь ожидает от других людей — не его личные дела |

### Атрибуты карточки

| Поле | Описание |
|---|---|
| `title` | Название |
| `description` | Описание |
| `size` | Размер в часах. `None` = 15 мин, `999` = от 1 часа и более (если осталось время после других задач) |
| `importance` | Важность: `среднее` / `важное` / `критическое` |
| `deadline` | Дедлайн (поле «Срок») |
| `event_time` | Дата и время события (кастомное поле `id_590358`) |
| `weekday` | День недели (кастомное поле `id_590359`): ПН–ВС — для еженедельных задач |
| `comments` | Список комментариев |
| `tags` | Метки (см. ниже) |

### Теги

| Тег | Смысл |
|---|---|
| `ежедневно` | Появляется каждый день |
| `по будням` | Появляется по будним дням (пн–пт) |
| `по выходным` | Появляется в субботу и воскресенье |
| `еженедельно` | Появляется раз в неделю в день, указанный в поле `weekday` |
| `напомнить` | Включает напоминание перед событием |
| `вечерняя` | Помещает задачу в вечерний блок 19:00–22:00 |

## 2. Уведомления в Telegram

Работают только для карточек с тегом `напомнить` и заполненным `event_time`:

- За **15 минут** до события (важность «среднее» или не указана)
- За **30 минут** и **15 минут** до события с важностью «важное» или «критическое» — с пометкой «ВАЖНО»

## 3. Команды в Telegram-боте

| Команда | Действие |
|---|---|
| `утро` | Запустить утреннюю логику, прислать план-расписание + кнопки карточек, закрепить сообщение с планом |
| `пересобрать` / `перепланируй` | Пересобрать расписание с текущего момента: не трогает прошедшие/идущие задачи и жёсткие встречи (тег «жёсткое событие»), пересобирает только «мягкие» авторазмещённые карточки. Не сбрасывает флаг `morning_done` — доступна сколько угодно раз за день |
| `вечер` | Подвести итог дня (временно отключена) |
| `создать [описание]` | Создать карточку в нужной колонке. Поддерживает явный размер: «размер - 2», «размер 0,5» → часы (дробное число) |
| `готово [описание]` | Для обычных задач — переместить в Архив с комментарием «Выполнено {дата}». Для регулярных задач (тег `ежедневно`/`по будням`/`по выходным`/`еженедельно`) — НЕ архивирует, а переносит на следующий подходящий день по тегу, чтобы не терять ротацию |
| `перенести [описание] [куда]` | Переместить карточку. Перенос критической задачи (`важность=критическое`) с дедлайном сегодня/завтра запрашивает подтверждение |
| `заметка [описание] // [текст]` | Добавить комментарий к карточке. Разделитель между названием и текстом — `//` |

Секция «На контроле» также распознаётся в бытовых формулировках («жду от Иванова счета», «ждём ответа от юристов» и т.п.), не только явным словом.

При переносе карточки в секцию «На контроле» (через `перенести`/диалог) автоматически добавляется комментарий «Задача переведена на контроль {дата}» — чтобы отслеживать, как долго задача там висит.

## 4. Интерактивные кнопки карточек

После команды «утро» (и после автоматической утренней джобы, и после «пересобрать») бот присылает
сообщение с кнопками — по одной на каждую карточку сегодняшнего дня, отсортированные по времени
размещения (совпадает с порядком в расписании). Постранично: максимум 20 кнопок на странице,
с навигацией «← Назад» / «Ещё →» если карточек больше.

При нажатии на карточку появляются кнопки действий:

| Кнопка | Действие |
|---|---|
| ✅ Готово | Запрашивает комментарий («.» — без комментария) → для регулярных задач переносит на следующий цикл, для обычных — архивирует |
| ⏭ Продолжить в другой день | Для коротких задач (`size ≤ 0.25`ч) переносит сразу без вопроса. Иначе запрашивает «Сколько часов реально нужно ещё?» → обновляет `size` → переносит на следующий подходящий день (регулярные — по метке, обычные — на завтра) |
| 💬 Комментарий | Запрашивает текст → добавляет комментарий в карточку |
| 📅 Перенести | Запрашивает куда («завтра», «в пятницу», «23 июня») с опциональным уточнением секции. Для критической задачи с дедлайном сегодня/завтра — запрашивает подтверждение переноса |
| 🤖 Совет | Запрашивает вопрос → отправляет в Claude API (вопрос + название + описание + все комментарии) → выводит ответ + сохраняет вопрос и ответ как комментарий в карточке |
| 🔔 Напоминалка | Запрашивает дату и время → устанавливает `event_time` → добавляет тег `напомнить` |
| ← Назад | Возврат к списку карточек без изменений |

После любого действия — подтверждение и обновлённый список карточек (первая страница).

### 4.1. Подтверждение риска переноса

Если карточка имеет `важность=критическое` и дедлайн сегодня или завтра — перенос (кнопкой
«Перенести» или «Продолжить в другой день») требует явного подтверждения («Да, перенести» /
«Отмена») перед выполнением. Автоматический перенос при overflow (Этап 4 утренней логики)
подтверждения не запрашивает — вместо этого такие карточки помечаются ⚠️ в отчёте об overflow.

## 5. Утренняя логика

Запускается автоматически в **06:30** по московскому времени или вручную командой «утро».
Повторный запуск в тот же день — пропускается (флаг в SQLite).

«Сегодня» везде в системе (утренняя логика, хендлеры бота, очистка Архива) вычисляется строго
по московскому времени (UTC+3), а не по системной дате сервера (Railway работает в UTC) — иначе
рядом с полуночью MSK день недели определяется неверно.

### 5.1. Сбор карточек в пул

#### Обычный день (вторник – воскресенье)

Берём все незавершённые карточки из **вчерашней колонки**, а также карточки из **сегодняшней колонки с устаревшей датой события** (`event_time.date() < сегодня` или `event_time is None`) — кроме разделителей (`blocked=True`) и карточек в секции «На контроле».

Карточки из сегодняшней колонки с `event_time.date() == сегодня` уже зарезервированы в планировщике (Фаза 0) и в пул не добавляются.

Обработка по типу:

| Тип карточки | Действие |
|---|---|
| Тег `еженедельно` + weekday раньше чем сегодняшний день| Переместить в «Следующая неделя», в пул не добавлять |
| Тег `еженедельно` + weekday позже чем сегодняшний день| Переместить в колонку дня по weekday, в пул не добавлять |
| Секция «На контроле» | Переместить в секцию «На контроле» сегодняшней колонки без назначения времени, в пул не добавлять |
| Тег `по будням` + сегодня суббота или воскресенье | Переместить в «Следующая неделя», в пул не добавлять |
| Тег `по выходным` + сегодня будний день | Переместить в колонку «Суббота», в пул не добавлять |
| Все остальные (включая `ежедневно`, `по будням` в будни, `по выходным` в выходные, `еженедельно` в день по weekday) | Добавить в пул для дальнейшего распределения |

#### Понедельник

Берём незавершённые карточки из **всех колонок пн–вс + «Следующая неделя»**.

Обработка по типу:

| Тип карточки | Действие |
|---|---|
| `event_time` указывает на конкретный день текущей недели | Переместить напрямую в колонку этого дня, секция по времени (`< 12:00` → Утро, `12:00–18:59` → День, `≥ 19:00` → Вечер), установить `event_time`. В пул не добавлять |
| Тег `еженедельно` + weekday не `ПН`| Переместить в колонку по weekday, в пул не добавлять |
| Секция «На контроле» | Переместить в «На контроле» понедельника без времени, в пул не добавлять |
| Тег `по будням` | Добавить в пул (понедельник — будний день) |
| Тег `по выходным` | Переместить в колонку «Суббота», в пул не добавлять |
| Все остальные | Добавить в пул |

После распределения пула: переместить часть карточек из «Далекие времена» в «Следующая неделя» (у которых `event_time` попадает на следующую неделю или нет конкретного срока, но пора планировать).

---

### 5.2. Этап 1 — Фиксированные события (Phase 1)

Из пула выделяем карточки у которых **`event_time.date() == сегодня`**, а также карточки с метками "ежедневно", "еженедельно", "по будням", "по выходным", у которых задано время у События.

Для каждой такой карточки:
- `event_time` - меняем только дату на сегодня; время **не меняем** — оставляем как есть
- Определяем секцию по часу события: `< 12:00` → Утро, `12:00–18:59` → День, `≥ 19:00` → Вечер
- Перемещаем в соответствующую секцию сегодняшней колонки
- Резервируем временной слот в планировщике (чтобы остальные задачи их огибали)

Оставшиеся карточки (без `event_time` на сегодня) — Этап 1б, затем Этап 2.

---

### 5.2б. Этап 1б — Карточки с будущей датой события

После Фазы 1 из оставшегося пула выделяем карточки у которых **`event_time.date() > сегодня`**.

Каждую такую карточку перемещаем в колонку соответствующую дате события:
- Дата на этой неделе (≤ воскресенье текущей недели) → колонка нужного дня (`WEEKDAY_COLUMNS[weekday]`)
- Дата на следующей неделе → «Следующая неделя»
- Дата ещё позже → «Далекие времена»

Секция внутри колонки — по времени события: `< 12:00` → Утро, `12:00–18:59` → День, `≥ 19:00` → Вечер.

В пул не добавляются — они уже там где надо.

---

### 5.3. Этап 2 — Классификация по приоритетам (10 групп)

Каждая оставшаяся карточка попадает в одну из 10 групп. Определения:
- `today_dl` = дедлайн == сегодня
- `soon_dl` = дедлайн == завтра или послезавтра
- `is_eve` = тег `вечерняя`

| Группа | Условие | Блок | Секция | Доп. сортировка |
|---|---|---|---|---|
| 0 | критическое + `today_dl` + не вечерняя | Рабочий | Утро | size ASC |
| 1 | важное + `today_dl` + не вечерняя | Рабочий | Утро | size ASC |
| 2 | критическое + `soon_dl` + не вечерняя | Рабочий | День | — |
| 3 | важное + `soon_dl` + не вечерняя | Рабочий | День | — |
| 4 | среднее + `today_dl` или `soon_dl` + не вечерняя | Рабочий | День | — |
| 5 | все остальные без тега `вечерняя` + size не `999` | Рабочий | День | — |
| 6 | все остальные без тега `вечерняя` + size `999` | Рабочий | День | — |
| 7 | вечерняя + критическое + `today_dl` | Вечерний | Вечер | size ASC |
| 8 | вечерняя + важное + `today_dl` | Вечерний | Вечер | size ASC |
| 9 | вечерняя + все остальные | Вечерний | Вечер | — |

---

### 5.4. Этап 3 — Назначение времени

**Два временных блока:**
- **Рабочий:** 09:00–19:00 — для групп 0–5
- **Вечерний:** 19:00–22:00 — для групп 7–9

Утро и День — ярлыки приоритета, не отдельные временные окна. Оба используют единый рабочий блок.

**Интерпретация поля `size`:**
- `None` (не указано) → 15 минут
- `999` → занять 1 час или всё оставшееся свободное время блока, если в пуле нет больше задач на этот блок
- Число → количество часов

**Алгоритм размещения (сегментированный):**

Для каждой карточки в порядке группы и сортировки:
1. Вычислить нужную длительность в минутах
2. Найти свободные интервалы в блоке (с учётом фазы 1 и уже размещённых задач)
3. Если суммарного свободного времени достаточно — размещаем:
   - Задача заполняет свободные интервалы последовательно, **огибая** фиксированные события
   - Например: задача на 2 ч, в блоке занято 10:00–11:00 → задача занимает 09:30–10:00 и 11:00–12:30
   - Ключевое правило: `chunk = min(нужно_ещё, размер_интервала)` — никогда не берём больше нужного
4. Устанавливаем `event_time` = начало первого сегмента (сегодняшняя дата + время)
5. Перемещаем карточку в соответствующую секцию
6. Если свободного времени не хватает → карточка уходит в переполнение (Этап 4)

**Тег «не дробить»:** карточка с этим тегом размещается только целиком, в один непрерывный
интервал (не дробится на части). Если целого интервала нужной длины нет — карточка уходит
в overflow, даже если суммарно свободного времени хватило бы по частям.

**Заполнение окон, пропущенных ранними группами:** атомарная («не дробить») или сегментированная
задача может «перепрыгнуть» через свободный промежуток, который слишком мал для неё, заняв более
поздний интервал. Чтобы этот пропущенный промежуток не пропадал навсегда, перед началом обработки
КАЖДОЙ следующей группы приоритета курсор блока сбрасывается к началу блока (occupied-список
при этом не сбрасывается и по-прежнему исключает наложение на уже размещённые задачи) — так более
поздние (менее приоритетные) группы могут заполнить оставшиеся пустые промежутки более ранних групп.
Внутри одной группы курсор продолжает двигаться вперёд как обычно (порядок карточек внутри группы
не меняется).

**Минимальный сегмент:** обычные (дробимые) задачи не создают сегментов короче 15 минут —
интервалы меньше 15 минут пропускаются при нарезке (кроме случая, когда единственный оставшийся
интервал в принципе короче 15 минут — тогда используется он, лучше короткий сегмент чем overflow).

---

### 5.5. Этап 4 — Переполнение

Карточки, которые не влезли сегодня:
- Перемещаются в **следующий день** (без назначения `event_time`)
- Если следующий день — понедельник следующей недели → перемещаются в «Следующая неделя»
- Если следующий день — суббота/воскресенье, а карточка помечена тегом «рабочая» — переносится
  сразу в «Следующая неделя» вместо выходного дня
- Следующим утром эти карточки будут распределены заново вместе с задачами того дня

**Прозрачность overflow:** каждый прогон (утренний и «пересобрать») формирует отчёт о перенесённых
карточках (название + куда перенесена), который добавляется в конец сообщения с планом —
«📤 Перенесено на другое время». Карточки с `важность=критическое` и дедлайном сегодня/завтра
помечаются в этом отчёте значком ⚠️.

---

### 5.6. Тег «жёсткое событие» и пересборка в течение дня

Тег «жёсткое событие» ставится карточкам с явным временем события как в Фазе 1 (карточки из пула),
так и в Фазе 0 (карточки, уже стоящие в сегодняшней колонке с `event_time.date() == сегодня`,
которые в пул не попадают, но резервируют слот) — маркер того, что это реальная встреча/
фиксированное время, а не авторазмещённая алгоритмом задача. При создании карточки командой чата
с явным временем на сегодня — тег также ставится, и бот предлагает пользователю пересобрать расписание.

Команда «пересобрать» (см. раздел 3) запускает `MorningLogic.replan(now)`:
1. Карточки «На контроле» и прошедшие/идущие (`event_time ≤ now`) — не трогаются.
2. Будущие карточки с тегом «жёсткое событие» — не двигаются, но резервируют слот в планировщике.
3. Остальные (кандидаты) — пересобираются по тем же 10 группам приоритета и алгоритму
   сегментированного размещения (Этапы 2–3), но курсор блоков стартует от текущего момента,
   а не от 09:00. Если сейчас позже 22:00 — все кандидаты уходят в overflow.

**Отображение отчёта после «пересобрать»:** сообщение с планом показывает только слоты начиная
с текущего момента — карточки с `event_time` в прошлом (уже прошедшие часы) из текста плана
исключаются, даже если сама карточка (прошедшая/идущая или жёсткая) не была затронута
пересборкой. Карточки без времени (например, «На контроле») показываются всегда.

Если карточка идёт прямо сейчас (начало ≤ текущий момент < конец, конец вычисляется из
сегментов либо `event_time + size`) — она не исключается, а показывается с обрезанным
началом: «текущий момент–конец» (например, задача 19:00–20:00 при пересборке в 19:27
отображается как «19:27–20:00»). Только полностью прошедшие задачи (конец ≤ текущий момент)
исключаются из отчёта целиком.

---

### 5.6. После распределения — отчёт Claude

После завершения всех перемещений:
1. Загрузить карточки сегодняшней колонки с комментариями (параллельно через `asyncio.gather`)
2. Отсортировать по `event_time`
3. Передать в Claude API с полями: название, описание, комментарии, важность, дедлайн, время события, длительность, сегменты (если задача разбита)
4. Claude генерирует хронологическое расписание с советами по каждой задаче
5. Текст отправляется в Telegram (разбивается на части если > 4096 символов)
6. Следом отправляется сообщение с кнопками-карточками

**Формат отчёта Claude:** хронологический список с временными метками.
Секция «На контроле» — отдельным блоком в конце, с советами по контролю исполнителей.
Прерываемые задачи показываются частями: `(часть 1/2)`, `(часть 2/2)`.

---

## 6. Вечерняя логика

Временно отключена. При команде «вечер» бот отвечает: «Вечерний отчёт временно недоступен».

Планируемая логика (следующая версия):
- Загрузить незавершённые карточки сегодняшней колонки
- Загрузить выполненные из Архива (по полю `updated_at` == сегодня)
- Claude строит честный итог: что сделано, что нет, урок дня, мотивация на завтра
- Секция «На контроле» оценивается отдельно — не личный провал, а ожидание от других

---

## 7. Очистка Архива

Каждый понедельник в **06:00** (до утренней логики, `misfire_grace_time=6ч` — не времязависима,
допуск шире чем у утренней джобы):
- Загрузить карточки из колонки Архив
- Удалить те у которых `last_moved_at` (дата последнего перемещения между колонками; если поле
  недоступно в ответе Kaiten API — fallback на `updated_at`) старше 30 дней
- Логировать количество удалённых и диагностику (сколько пропущено как заблокированные/без даты/
  слишком новые) — для отладки, если чистка неожиданно ничего не удалила

---

## 8. Автономность

Система работает без участия пользователя:
- 06:00 пн — очистка Архива
- 06:30 ежедневно — утренняя логика (если пользователь не написал «утро» раньше)
- Каждую минуту — проверка напоминалок
- Если пользователь написал «утро» раньше 06:30 — автозапуск в 06:30 пропускается (флаг в SQLite)

## 9. Кнопка «Совет»

При нажатии запрашивает вопрос пользователя. Claude получает:
- Вопрос
- Название карточки
- Описание карточки
- Все комментарии карточки (включая предыдущие вопросы и ответы)

Ответ выводится в чат. Вопрос и ответ сохраняются как комментарий в карточке:

❓ Вопрос: {вопрос}
💡 Ответ: {ответ}

## 13. Мульти-пользовательская поддержка

Сервис поддерживает нескольких пользователей. Каждый пользователь работает со своей доской Kaiten
и своим Telegram-чатом, независимо от других.

### 13.1. Конфигурация пользователей

Список пользователей задаётся в `users.json` (не в git):

```json
[
  {
    "user_id": "owner",
    "telegram_chat_id": 111111111,
    "kaiten_board_id": 476640,
    "kaiten_lane_id": 623640,
    "kaiten_space_id": 197396,
    "column_ids": {
      "Понедельник": 1688101,
      "Вторник": 1689798
    }
  },
  {
    "user_id": "alice",
    "telegram_chat_id": 222222222,
    "kaiten_board_id": 999999,
    "kaiten_lane_id": 0,
    "kaiten_space_id": 197396,
    "kaiten_token_env": "KAITEN_TOKEN_ALICE",
    "kaiten_base_url_env": "KAITEN_BASE_URL_ALICE"
  }
]
```

Если `users.json` отсутствует — сервис читает конфиг из env-переменных (обратная совместимость):
`TELEGRAM_CHAT_ID`, `KAITEN_BOARD_ID`, `KAITEN_LANE_ID`, `KAITEN_SPACE_ID`.

**Токены Kaiten:** хранятся в env-переменных Railway, не в `users.json`.
В JSON прописывается только имя переменной (`kaiten_token_env`). Если поле отсутствует —
используется глобальный `KAITEN_TOKEN` из env (для пользователей на одном аккаунте).

### 13.2. Изоляция пользователей

Для каждого пользователя создаются отдельные экземпляры:
- `KaitenClient(board_id, lane_id, ...)` — работает только со своей доской
- `BoardLogic(client, column_ids)` — хранит маппинг колонок
- `MorningLogic(client, logic)` — работает с колонками пользователя
- `Notifier(chat_id)` — отправляет только в чат пользователя, умеет закреплять сообщение
  с планом (открепляя предыдущие) и слать кнопки карточек без входящего Update

### 13.2.1. Разные ID тегов/полей на разных Kaiten-аккаунтах

Если у пользователя другой Kaiten-аккаунт — ID тегов регулярности и ID кастомных полей
(«Событие»/«Важность»/«День недели») почти наверняка другие, а сами кастомные поля могут
отсутствовать вовсе (Kaiten API не поддерживает их создание через API — нужно создать
вручную в Kaiten UI при онбординге). `KaitenClient` принимает опциональные оверрайды
(`tag_ids`, `importance_options`, `weekday_options`, `field_ids`), задаваемые per-user в
`users.json`; без них используются значения по умолчанию (текущий основной аккаунт).
Подробности и пример JSON — `docs/interfaces.md`, раздел «Мультиаккаунт Kaiten».

Роутинг в Telegram по `chat_id`: `HandlersConfig.users[chat_id]` → `UserHandlerCtx`.
Неизвестный `chat_id` — игнорируется без ответа.

### 13.3. Расписание

Утренняя логика (06:30) запускается для **всех** пользователей последовательно.
Напоминалки — для каждого пользователя независимо (проверяются все каждую минуту).

## 14. Настройка доски (board_setup)

При первом запуске для нового пользователя (или если `column_ids` не заданы в users.json)
сервис автоматически настраивает доску Kaiten.

### 14.1. Создание колонок

Создаются стандартные колонки если отсутствуют:
```
Понедельник, Вторник, Среда, Четверг, Пятница, Суббота, Воскресенье,
Следующая неделя, Далекие времена, Долгий ящик, Архив
```

Alias-маппинг: если на доске есть "Пн" — он принимается как "Понедельник" и т.д.

### 14.2. Создание разделителей

В каждой колонке дня (Пн–Вс) создаются разделители-карточки:
- `title = "-"`, `blocked = True`
- `block_reason` = "Утро" / "День" / "Вечер" / "На контроле"

В "Следующая неделя" разделители не создаются.

### 14.3. Идемпотентность

- Существующие колонки и разделители не пересоздаются
- Пустые лишние колонки — удаляются
- Непустые лишние колонки — не трогаются (предупреждение в логах)
- Если `column_ids` заданы в users.json — board_setup полностью пропускается
