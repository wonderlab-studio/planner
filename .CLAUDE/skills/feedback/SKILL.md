# Skill: Feedback Simulation — AI Personal Planner

Симулирует использование сервиса-планировщика параллельными воображаемыми пользователями
(персонами) на заданном отрезке времени и собирает от них продуктовую обратную связь:
что полезно, что неудобно, чего не хватает. Работает по треку **Продукт**.

Вся симуляция — текстовая, без реальных вызовов Kaiten/Telegram/Claude API. Источник
истины о механике сервиса — `docs/requirements.md`.

---

## Входные параметры

Пользователь при вызове скилла указывает:

- **N** — сколько профилей-персон запускать параллельно
- **Длительность симуляции** — период, который проживает каждая персона
  (например «3 дня», «1 неделя», «2 недели», «1 месяц»)

Если хотя бы один параметр не указан явно в запросе — **спросить пользователя перед
запуском**, не подставлять дефолты. N и длительность напрямую определяют объём работы
и стоимость (N параллельных Fable-агентов × токены на промпт).

---

## Шаг 0 — Подготовка

1. Прочитать `docs/requirements.md` — механика сервиса (алгоритм утра, теги, секции,
   кнопки, команды, уведомления). Это нужно, чтобы адекватно подобрать персон и позже
   осмысленно собрать сводный отчёт — сами персона-агенты тоже читают этот файл
   самостоятельно.
2. Прочитать `.claude/skills/feedback/personas.md` — банк архетипов ЦА.
3. Взять сегодняшнюю дату из контекста сессии — симуляция стартует с неё. Вычислить
   конечную дату периода и по возможности разметить конкретные календарные дни (с
   днями недели) — персона-агентам нужны реальные даты, чтобы корректно проверить
   недельную логику (понедельничный пересбор, «еженедельно», «по выходным» и т.д.).
4. Создать директорию запуска отчёта:
   `docs/feedback-reports/{YYYY-MM-DD}-{N}p-{duration-slug}/`
   например `docs/feedback-reports/2026-07-07-5p-1week/`.
   Внутри будет `report.md` (сводный) и `diary-{slug-персоны}.md` на каждую персону.

---

## Шаг 1 — Подбор персон

- Выбрать N архетипов из `personas.md`, максимизируя разнообразие категорий (не брать
  два похожих профиля, если есть из чего выбрать непохожие).
- Если N больше числа архетипов в банке — дополнить новыми в том же формате
  (демография / занятость / привычки / потребности), и предложить пользователю
  дописать их в `personas.md` для будущих запусков (не делать это без подтверждения).
- Каждой выбранной персоне присвоить короткий slug (латиницей, для имён файлов),
  например `freelance-designer`, `pm-corporate`, `student-parttime`.

---

## Шаг 2 — Параллельный запуск персон-субагентов

Все N вызовов `Agent(subagent_type="persona-agent")` — **одним сообщением**, несколько
tool calls параллельно. Персоны не зависят друг от друга и не должны знать друг о друге.

### Шаблон промпта для каждой персоны

```
Ты симулируешь использование AI-планировщика персоной из категории: {архетип из personas.md
— вставить весь блок целиком: демография/занятость/привычки/потребности}.

Придумай на основе этого архетипа конкретного человека (имя, возраст, детали быта) и
проживи с ним/с ней период симуляции: с {start_date} ({день недели}) по {end_date}
({день недели}) включительно.

Сегодняшняя дата в реальном мире (для справки, не используется в симуляции): {today}.

Сохрани дневник в файл: {run_dir}/diary-{slug}.md

Верни финальный ответ в формате, описанном в твоей роли (Портрет / Что оказалось полезным /
Трудности / Пожелания / Итоговое отношение).
```

Не указывать `run_in_background: false` — агенты выполняются в фоне параллельно,
оркестратор дождётся уведомлений по мере готовности каждого (не опрашивать, не спать).

---

## Шаг 3 — Сбор результатов и сводный отчёт

Дождаться завершения всех N агентов. Как только все ответы получены — собрать
`report.md` в директории запуска.

### Структура `report.md`

```markdown
# Отчёт по симуляции — {N} персон, период {start_date}–{end_date}

Сгенерировано: {today}

## Сквозные наблюдения
- Что хвалят почти все персоны
- Топ-3 повторяющиеся боли (across персон)
- Топ-3 самых частых/весомых пожелания
- Есть ли противоречащие друг другу пожелания разных сегментов (стоит явно отметить —
  продукту придётся выбирать приоритет)

## Персона: {имя} — {архетип}
[вставить структурированный ответ агента целиком]
Дневник: `diary-{slug}.md`

## Персона: ...
...
```

Раздел «Сквозные наблюдения» пишет сам оркестратор после прочтения всех N ответов —
это главная ценность сводного отчёта, не просто конкатенация персон.

---

## Шаг 4 — Резюме пользователю

Кратко (до ~200 слов): сквозные паттерны из «Сквозных наблюдений» и путь к
`docs/feedback-reports/{run_dir}/report.md`. Не пересказывать все персоны в чат —
отчёт для этого и сохранён.

---

## Важные ограничения симуляции

- Всё происходит в рассуждении persona-agent'ов. Реальные вызовы Kaiten API,
  Telegram API, Claude API — не производятся, никакие внешние MCP/Bash-инструменты
  для этого не используются.
- Работа каждого субагента должна занимать заметно меньше времени, чем моделируемый
  период — дни симуляции проживаются последовательно в рамках одного хода
  рассуждений агента, без реального ожидания (никаких ScheduleWakeup между "днями").
- Механика должна строго соответствовать `docs/requirements.md`. Если персона хочет
  то, чего сервис не умеет — это находка для отчёта, а не повод придумать
  несуществующую функциональность.
- Персона-агенты используют модель Fable (`model: claude-fable-5`, задано в
  `.claude/agents/persona-agent.md`) — она даёт более живые, характерные портреты,
  чем стандартная модель разработки.
